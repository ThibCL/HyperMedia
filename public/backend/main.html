Specification

Web Architecture

To ensure that the HTML is not rendered server side we used the package "serve-static" of npm that permits to send to the browser static file. Then the browser renders the HTML and to change content depending on the user request we do ajax call to the API.

API

REST compliance

We have followed up completely the REST principles by using the HTTP verbs to do the request (we actually used only GET because that was the only one needed), and by using the right scheme for requests to describe the ressources wanted. For example we used the path to specify the service to use and some other parameter related to the service, and we used query parameter for complementary data not related to the service specify in the path.

OpenAPI Resource models

For ressources in the OpenAPI specification we have introduced models for the three main object in our application (Service, Person, Event) with all the information stored in the database. But also other models that correspond to response of some requests, for example ServiceDescripion that contains only main information to describe the service in the page "All service". The purpose is to limit the information send back to the minimum needed by the request.

Data model

These ER diagram map to the OpenAPI data model for the main objects (Service, Event, Person). But the database store also the relation between them (involves, presents).

Implementation

Tools used

For the backend of the application we used the language Javascript with the environment NodeJs to help implementing the server. Moreover we used the library 'knex.js' for the SQL queries. For the tools we used VsCode as text editor that is really helpful with the different extensions and also Github to be able to develop the application at two and review the work of the other one.

Discussion

Describe here:

  • We used the swagger ui tool to test that the web application adheres to the OpenAPI specification. Indeed we used the feature "Try it out" to check that the response provided by the application correspond to the specification.
  • Our web application adheres to common practices to partition a REST-based web application because we separate the static assets (bootstrap, css html) and the data. In fact the API can be used without the front to retrieve all the information about the association and after we used these information to populate the html pages.
  • For our qpplication we decided to not do it because it was not necessary to connect. So all the endpoints are Get requert to retrieve data from the back.
  • We decided to use a relationnal database since the schema of our data is static and not complicated.(see the diagram above)
